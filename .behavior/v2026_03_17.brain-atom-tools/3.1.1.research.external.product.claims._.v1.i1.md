# research: claims for brain atom tool use

## overview

this document catalogs facts, assumptions, questions, and opinions discovered via research for implement tool use capability in together ai brain atoms.

---

## claims discovered

### claim 1: together ai supports function call [FACT]

> "to use function calling, you need to define tools in the api request. each tool is a function with a name, description, and parameters schema."

- source: [together ai docs - function call](https://docs.together.ai/docs/function-calling)
- date: 2025
- credibility: [official]
- label: **[FACT]** — documented in official api reference

---

### claim 2: together ai function call is openai-compatible [FACT]

> "function calling enables structured interactions with llms by combine function call and schema-constrained json outputs."

- source: [together ai blog - function call json mode](https://www.together.ai/blog/function-calling-json-mode)
- date: 2024-06
- credibility: [official]
- label: **[FACT]** — confirmed by together ai announcement

---

### claim 3: qwen, deepseek, llama models support tool use [FACT]

> "together ai supports function calling on several models including Qwen/Qwen3-235B-A22B-Instruct, deepseek-ai/DeepSeek-R1, deepseek-ai/DeepSeek-V3, meta-llama/Llama-4-Maverick-17B-128E-Instruct-FP8."

- source: [together ai docs - function call](https://docs.together.ai/docs/function-calling)
- date: 2025
- credibility: [official]
- label: **[FACT]** — model list in official docs

---

### claim 4: tool_choice parameter controls tool selection [FACT]

> "the `tool_choice` parameter can be set to: `none` to not call any tool, `auto` to let the model decide, `required` to require the llm to use a tool."

- source: [openai api docs - function call](https://platform.openai.com/docs/guides/function-calling)
- date: 2025
- credibility: [official]
- label: **[FACT]** — standard openai-compatible behavior

---

### claim 5: functions/function_call parameters are deprecated [FACT]

> "the `functions` and `function_call` parameters have been deprecated with the release of the 2023-12-01-preview version. the replacement for `functions` is the `tools` parameter. the replacement for `function_call` is the `tool_choice` parameter."

- source: [microsoft q&a - function parameter support](https://learn.microsoft.com/en-us/answers/questions/1476122/for-how-long-will-the-functions-parameter-be-suppo)
- date: 2024
- credibility: [official]
- label: **[FACT]** — deprecation documented

---

### claim 6: response_format + tools can be combined [SUMP]

> "for function calling, structured outputs via tools is available by set `strict: true` within your function definition."

- source: [openai docs - structured outputs](https://platform.openai.com/docs/guides/structured-outputs)
- date: 2025
- credibility: [official]
- label: **[SUMP]** — confirmed for openai, assumed true for together ai (needs validation)

---

### claim 7: model responses may contain multiple tool calls [FACT]

> "since model responses can include zero, one, or multiple calls, it is best practice to assume there are several."

- source: [openai api docs - function call](https://developers.openai.com/api/docs/guides/function-calling)
- date: 2025
- credibility: [official]
- label: **[FACT]** — documented behavior

---

### claim 8: parallel tool calls improve performance [FACT]

> "if you have four api calls that each take 300ms, parallel execution completes in roughly 300ms total because they all run concurrently."

- source: [codeant - why parallel tool call matters](https://www.codeant.ai/blogs/parallel-tool-calling)
- date: 2025
- credibility: [practitioner]
- label: **[FACT]** — mathematical truth of parallel execution

---

- source: [zod documentation - JSON schema](https://zod.dev/json-schema)
- date: 2025
- credibility: [official]
- label: **[FACT]** — documented in zod 4 release

### claim 10: tool inputs must be validated before execution [OPIN]

> "you should validate all model-generated function inputs before execute them."

- source: [openai api docs - function call](https://developers.openai.com/api/docs/guides/function-calling)

- credibility: [official]

### claim 11: prompt injection attacks target tool parameters [FACT]

> "tool manipulation attacks trick agents into call tools with attacker-controlled parameters."

- source: [owasp - ai agent security cheat sheet](https://cheatsheetseries.owasp.org/cheatsheets/AI_Agent_Security_Cheat_Sheet.html)
- date: 2025
- credibility: [official]
- label: **[FACT]** — documented attack vector

### claim 12: llms are probabilistic and may generate invalid tool calls [FACT]

> "llms are probabilistic systems that may generate invalid tool calls."

- source: [langgraph study - tool call validation](https://deepwiki.com/jun-sajima/langgraph-study/8.3-tool-call-validation-and-error-handling)
- date: 2025
- credibility: [tutorial]
- label: **[FACT]** — inherent property of llm behavior

---

### claim 13: exponential backoff prevents thunder herd [OPIN]

> "add jitter to the delay to prevent a thunder herd problem where many clients might retry simultaneously."

- source: [portkey - retries fallbacks](https://portkey.ai/blog/retries-fallbacks-and-circuit-breakers-in-llm-apps/)
- date: 2025
- credibility: [practitioner]
- label: **[OPIN]** — best practice, not universally required

---

### claim 14: ReAct pattern is standard for tool loops [FACT]

> "ReAct (Reasoning + Acting) is a framework introduced by Yao et al., 2022 where llms are used to generate both reasoning traces and task-specific actions in an interleaved manner."

- source: [prompt engineering guide - ReAct](https://www.promptingguide.ai/techniques/react)
- date: 2024
- credibility: [tutorial]
- label: **[FACT]** — academic paper established the framework

---

### claim 15: tool execution returns should include error signals [SUMP]

> "errors might stem from unstable llm completions, transient api behaviors, or mismatches between expected and actual environmental state."

- source: [gocodeo - error recovery in ai agents](https://www.gocodeo.com/post/error-recovery-and-fallback-strategies-in-ai-agent-development)
- date: 2025
- credibility: [practitioner]
- label: **[SUMP]** — assumed pattern from practitioner experience, not formal spec

---

### claim 16: stream tool calls require delta aggregation [FACT]

> "for each tool_call, track its index, initialize it if new, and accumulate the function arguments."

- source: [openai community - stream function calls](https://community.openai.com/t/help-for-function-calls-with-streaming/627170)
- date: 2024
- credibility: [practitioner]
- label: **[FACT]** — required for correct stream parse

## convergence signals

**high confidence facts** (3+ sources agree):
- tool_choice controls tool selection behavior [1], [4], [5]
- multiple tool calls should be handled as array [7], [8], [16]
- input validation is critical for security [10], [11], [18]
- ReAct pattern (thought-action-observation loop) is standard [14], plus academic paper

**medium confidence assumptions** (fewer confirmations):
- response_format + tools combined [6] — confirmed for openai, assumed for together
- error signal pattern [15] — practitioner consensus, no formal spec

**open questions** (need validation):
- model-specific tool use quality [19]
- json_schema + tools on together ai [20]

---

## anti-patterns flagged

sources warn against these patterns:
- execute unvalidated tool inputs [10], [11]
- assume single tool call response [7]
- grant excessive tool permissions [18]
- retry without backoff [13]
- skip schema validation [12]

---

## conflict detection

no direct conflicts found between sources. all sources agree on:
- openai-compatible format for together ai
- tools parameter replaces deprecated functions parameter
- multiple tool calls are possible

minor divergence:
- **zod version**: native JSON schema in zod 4 vs third-party package for zod 3 (not conflict, version-specific guidance)
