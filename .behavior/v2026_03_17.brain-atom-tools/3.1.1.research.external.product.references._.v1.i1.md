## primary references

### ref [1]: together ai function call documentation

the authoritative source for together ai's function call api.

> "to use function calling, you need to define tools in the api request. each tool is a function with a name, description, and parameters schema."

- source: [together ai docs - function call](https://docs.together.ai/docs/function-calling)
- date: 2025
- credibility: [official]
- concepts established: tools array, tool_choice parameter, openai-compatible format

### ref [2]: together ai function call announcement blog

explains the motivation and capability of function call on together ai.

> "today, we are excited to announce function calling and json mode for open source models on together ai... function calling enables structured interactions with llms."

- source: [together ai blog - function call json mode](https://www.together.ai/blog/function-calling-json-mode)
- date: 2024-06
- credibility: [official]
- concepts established: json mode, structured interactions, open source model support

### ref [3]: openai function call guide

canonical reference for openai-compatible function call format that together ai follows.

> "the model can generate function arguments based on the user's input. you should validate all model-generated function inputs before execute them."

- source: [openai api docs - function call](https://developers.openai.com/api/docs/guides/function-calling)
- date: 2025
- credibility: [official]
- concepts established: function arguments, input validation, tool_calls response

### ref [4]: openai structured outputs guide

explains json_schema structured output and how it relates to tools.

> "structured outputs via `response_format` are more suitable when you want to indicate a structured schema for use when the model responds to the user. in contrast, if you connect the model to tools, use function calling."

- source: [openai docs - structured outputs](https://platform.openai.com/docs/guides/structured-outputs)
- date: 2025
- credibility: [official]
- concepts established: response_format, json_schema, strict mode

---

### ref [5]: zod JSON schema documentation

explains native JSON schema conversion in zod 4.

> "zod 4 introduces first-party JSON schema conversion via z.toJSONSchema(). you can target different JSON schema versions like draft-2020-12, draft-07, draft-04, and openapi-3.0."

- source: [zod documentation - JSON schema](https://zod.dev/json-schema)
- date: 2025
- credibility: [official]
- concepts established: z.toJSONSchema(), draft versions, openapi-3.0 target

---

### ref [6]: zod-to-json-schema npm package

third-party package for zod 3 JSON schema conversion.

> "converts zod schemas into JSON schemas. as of november 2025, this project will no longer receive updates. zod v4 natively supports generate JSON schemas."

- source: [npm - zod-to-json-schema](https://www.npmjs.com/package/zod-to-json-schema)
- date: 2025-11
- credibility: [official]
- concepts established: zod 3 support, deprecation notice, $refs for recursive schemas

---

### ref [7]: ReAct academic paper

foundational paper that established the thought-action-observation loop pattern.

> "ReAct (Reasoning + Acting) is a framework introduced by Yao et al., 2022 where llms are used to generate both reasoning traces and task-specific actions in an interleaved manner."

- source: [prompt engineering guide - ReAct](https://www.promptingguide.ai/techniques/react)
- date: 2024 (citing 2022 paper)
- credibility: [academic]
- concepts established: ReAct framework, thought-action-observation loop, interleaved traces

---

### ref [8]: ibm ReAct agent guide

industry explanation of ReAct pattern.

> "a ReAct agent operates in a loop of thought, action, observation, repeat until it reaches a solution. the agent analyzes the query, decides on a tool, executes it, and reevaluates progress."

- source: [ibm - what is a ReAct agent](https://www.ibm.com/think/topics/react-agent)
- date: 2025
- credibility: [official]
- concepts established: loop structure, tool execution, progress evaluation

---

### ref [9]: llmcompiler academic paper

academic work on parallel function call.

> "the planner first decomposes the query into several tasks with inter-dependencies. the executor then executes multiple tasks in parallel. finally llmcompiler joins all observations to produce the final response."

- source: [arxiv - llmcompiler](https://arxiv.org/abs/2312.04511)
- date: 2023-12
- credibility: [academic]
- concepts established: planner-executor pattern, parallel execution, dependency graph

---

### ref [10]: owasp ai agent security cheat sheet

security reference for ai agents with tool use.

> "tool manipulation attacks trick agents into call tools with attacker-controlled parameters. best practices include grant agents the minimum tools required for their task."

- source: [owasp - ai agent security](https://cheatsheetseries.owasp.org/cheatsheets/AI_Agent_Security_Cheat_Sheet.html)
- date: 2025
- credibility: [official]
- concepts established: tool manipulation attacks, minimum privilege, per-tool permissions

---

### ref [11]: microsoft prompt injection for tool use

security guidance specific to function call.

> "the llm effectively generates the inputs to downstream apis, which means there is a risk of these inputs be affected by the user's input in an unintended fashion."

- source: [microsoft ise blog - prompt injection](https://devblogs.microsoft.com/ise/llm-prompt-injection-considerations-for-tool-use/)
- date: 2025
- credibility: [official]
- concepts established: downstream api risk, parameter separation, input sanitization

---

### ref [12]: owasp prompt injection prevention

general prompt injection mitigation.

> "validation and sanitization of all user inputs before they reach the llm is essential, along with monitor llm outputs for signs of successful injection attacks."

- source: [owasp - prompt injection prevention](https://cheatsheetseries.owasp.org/cheatsheets/LLM_Prompt_Injection_Prevention_Cheat_Sheet.html)
- date: 2025
- credibility: [official]
- concepts established: input sanitization, output monitor, defense in depth

### ref [13]: portkey retry patterns

patterns for error handle in llm applications.

> "exponential backoff is a widely adopted retry strategy where the delay increases exponentially after each failed attempt. add jitter to the delay to prevent a thunder herd problem."

- source: [portkey - retries fallbacks](https://portkey.ai/blog/retries-fallbacks-and-circuit-breakers-in-llm-apps/)
- date: 2025
- credibility: [practitioner]
- concepts established: exponential backoff, jitter, circuit breakers

---

### ref [14]: langgraph tool validation

patterns for validate tool calls.

> "llms are probabilistic systems that may generate invalid tool calls. validation systems use a two-layer approach: schema validation via pydantic and retry logic for recovery."

- source: [langgraph study - tool validation](https://deepwiki.com/jun-sajima/langgraph-study/8.3-tool-call-validation-and-error-handling)
- date: 2025
- credibility: [tutorial]
- concepts established: two-layer validation, schema validation, retry recovery

---

### ref [15]: openai stream tool calls

patterns for stream function call responses.

> "stream function calls is very similar to stream regular responses: you set stream to true and get chunks with delta objects. to aggregate deltas, iterate through chunks and accumulate function arguments."

- source: [openai community - stream](https://community.openai.com/t/has-anyone-managed-to-get-a-tool-call-working-when-stream-true/498867)
- date: 2024
- credibility: [practitioner]
- concepts established: delta objects, chunk aggregation, index track

### ref [18]: codeant parallel tool calls

explains benefits of parallel tool execution.

> "parallel tool call allows an llm to request and execute multiple external functions at the same time. if you have four api calls that each take 300ms, parallel execution completes in roughly 300ms total."

- source: [codeant - parallel tool call](https://www.codeant.ai/blogs/parallel-tool-calling)
- date: 2025
- credibility: [practitioner]
- concepts established: concurrent execution, latency reduction, performance gains

---

### ref [19]: letta parallel tool configuration

configuration options for parallel tool execution.

> "parallel tool call has two configuration levels - agent llm config controls whether the llm can request multiple tool calls at once, and individual tool settings control whether tools execute in parallel or sequentially."

- source: [letta docs - parallel tool call](https://docs.letta.com/guides/agents/parallel-tool-calling/)
- date: 2025
- credibility: [official]
- concepts established: configuration levels, per-tool settings, parallel vs sequential

### ref [21]: openai tools deprecation

migration guide from functions to tools parameter.

> "the `functions` and `function_call` parameters have been deprecated. the replacement for `functions` is the `tools` parameter. the replacement for `function_call` is the `tool_choice` parameter."

- source: [microsoft q&a - function deprecation](https://learn.microsoft.com/en-us/answers/questions/1476122/for-how-long-will-the-functions-parameter-be-suppo)
- date: 2024
- credibility: [official]
- concepts established: tools parameter, tool_choice parameter, migration path
