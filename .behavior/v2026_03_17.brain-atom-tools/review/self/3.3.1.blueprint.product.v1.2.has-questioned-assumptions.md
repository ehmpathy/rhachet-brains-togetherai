## assumption 1: together ai uses openai-compatible format

**the assumption**: together ai function call api matches openai's chat completion format exactly.

- together ai docs explicitly state "OpenAI-compatible API" at docs.together.ai
- tested with real api calls — api accepts ChatCompletionTool format

**what if opposite were true**:

**exceptions found**: none. format matches exactly.

**counterexamples**: some providers (anthropic, google) use different formats. together chose openai compatibility.

**verdict**: ✅ holds — api is openai-compatible. no custom format needed.

## assumption 3: response_format and tools can coexist

**the assumption**: can send both response_format: json_schema AND tools parameter together.

**ISSUE FOUND in implementation**:
- when both present on initial invocation, model prioritizes json_schema
- model returns direct JSON answer instead of tool_calls
- this defeats the purpose of tool use

**root cause analysis**:
- response_format tells model "output must match this schema"
- tools tell model "you may call these functions"
- model interprets json_schema as stronger constraint than tool availability
- result: model answers directly in JSON, never invokes tools

**fix applied**:
```
wantStructuredOutput = !hasTools || promptIsToolExecutions
```
- initial call with tools: no response_format (allow tool calls)
- continuation after tools: yes response_format (want structured final answer)

**what if opposite approach**:
- could use tool_choice: "required" to force tool use? no — want model to decide
- could use "auto" and hope? no — model still prioritizes json_schema

**verdict**: ⚠️ issue found and fixed. response_format conditional on phase of interaction.

## assumption 5: three cast functions decomposition

**the assumption**: separate files for castIntoToolDef, castFromToolCall, castIntoToolMessages.

**evidence gathered**:
- matches rhachet-brains-xai implementation pattern
- each function handles one direction of data flow:
  - into api: BrainPlugToolDefinition → OpenAI.ChatCompletionTool
  - from response: tool_calls → BrainPlugToolInvocation
  - back to messages: BrainPlugToolExecution → tool messages

**could simpler approach work**:
- single file? would mix three unrelated transformations
- inline in genBrainAtom? would bloat the main file
- combine into/from? different data structures, different responsibilities

**architecture choice based on**:
- established pattern (xai, other brain suppliers)
- single responsibility principle
- testability of each cast function independently

**verdict**: ✅ holds — three files match data flow directions. follows established pattern.
