## deviations found

### deviation 1: prompt type is unconditional

**vision spec**:
```ts
prompt: TPlugs extends { tools: unknown[] }
  ? string | BrainPlugToolExecution[]
  : string;
```

**blueprint implementation**:
```ts
prompt: string | BrainPlugToolExecution[];
```

**analysis**:
this is a safe deviation. the unconditional union is MORE permissive than the conditional type. callers who pass strings still work. the deviation was noted in backcompat review and justified:
- simpler type definition
- rhachet BrainAtom interface may not support conditional types
- runtime behavior unchanged (Array.isArray check)

**verdict**: acceptable deviation — simpler, backwards compatible.

---

## why it holds

### contract adherance holds because:
- plugs.tools array matches vision exactly — BrainPlugToolDefinition with slug, name, description, schema
- prompt type accepts both string and BrainPlugToolExecution[] — allows initial prompt and continuation
- output type includes null case — correctly models "brain defers to tools"
- calls type includes null case — correctly models "brain answers directly"

### behavior adherance holds because:
- the response parse logic correctly branches: tool_calls present → output null, calls defined; no tool_calls → output defined, calls null
- castFromTogetherToolCall preserves provider's id as exid for correlation back to tool execution
- episode accumulation is unchanged from prior implementation — tool exchanges simply add to exchanges array

### acceptable deviation justified because:
- the unconditional union `string | BrainPlugToolExecution[]` is strictly more permissive than the conditional type
- callers who pass only strings (no tools plugged) still work — no breakage
- runtime behavior unchanged — Array.isArray check handles both cases
- simpler type is easier to understand and maintain
