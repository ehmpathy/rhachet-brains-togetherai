# self-review: 1.vision.has-questioned-requirements

## requirements questioned

### requirement 1: tool invocation with output schema

**who said this was needed?**
- the wish references xai pr #15 which implemented this pattern
- rhachet's BrainAtom contract requires `schema: { output }` — this is not optional

**what evidence supports this requirement?**
- xai implemented json_schema + tools together (confirmed in their tests)
- together ai docs confirm function call + response_format compatibility
- consistent contract across providers is a core rhachet value

**what if we didn't do this?**
- users would have to manually handle tool responses without structured output
- inconsistent experience vs xai/openai brain providers

**could we achieve the goal in a simpler way?**
- no — this is the minimal required feature for tool use parity

**verdict**: ✅ HOLDS — required for parity with xai implementation

---

### requirement 2: tool invocation without output schema

**who said this was needed?**
- listed in xai vision as a separate usecase

**what evidence supports this requirement?**
- some models may not support json_schema + tools together
- provides flexibility for free-form responses with tool results

**what if we didn't do this?**
- users would be forced to use json_schema even when not needed
- could cause errors on models without json_schema support

**could we achieve the goal in a simpler way?**
- this is already simple — just don't send response_format

**verdict**: ✅ HOLDS — provides flexibility for model compatibility

---

### requirement 3: tool continuation

**who said this was needed?**
- the wish explicitly says "same tool use extension and same tool use test coverage"
- xai pr #15 has tool continuation tests

**what evidence supports this requirement?**
- tool use is useless without continuation — brain requests tools, caller executes, brain synthesizes
- this is the core agentic loop pattern

**what if we didn't do this?**
- tool use would be broken — one-shot tool request with no way to feed results back

**could we achieve the goal in a simpler way?**
- no — this is the minimum viable tool loop

**verdict**: ✅ HOLDS — essential for tool use to function

---

### requirement 4: O(1) tool lookup dict

**who said this was needed?**
- mentioned in vision before/after contrast
- xai vision mentions `asBrainPlugToolDict`

**what evidence supports this requirement?**
- convenience utility for common pattern
- already exists in rhachet — just need to show usage

**what if we didn't do this?**
- users would do linear search or build their own dict
- not a blocker, but reduces pit-of-success

**could we achieve the goal in a simpler way?**
- this is provided by rhachet, not what we implement — just show usage

**verdict**: ✅ HOLDS — but this is rhachet's responsibility, not ours

---

### requirement 5: parallel tool call tests

**who said this was needed?**
- vision marks this as "out of scope for this work"

**what evidence supports this requirement?**
- wish says "same tool use test coverage" as xai
- xai pr #15 does NOT have explicit parallel tool call tests

**what if we didn't do this?**
- no impact — together ai handles parallel calls by default, no special code needed

**could we achieve the goal in a simpler way?**
- correctly scoped as out-of-scope in vision

**verdict**: ✅ CORRECTLY EXCLUDED — not in xai pr scope, not needed here
