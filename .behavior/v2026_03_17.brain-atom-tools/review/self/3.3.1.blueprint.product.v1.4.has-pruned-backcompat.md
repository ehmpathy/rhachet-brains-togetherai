### genBrainAtom extension — additive changes

| change | prior behavior | new behavior | verdict |
|--------|---------------|--------------|---------|
| TPlugs generic | no generic | TPlugs extends BrainPlugs = BrainPlugs | ✅ default preserves prior |
| prompt type | string only | string \| BrainPlugToolExecution[] | ⚠️ broader type — see analysis |
| output.calls | undefined | { tools: [] } \| null | ✅ additive property |

**detailed analysis of prompt type change**:

the blueprint specified conditional prompt type:
```ts
prompt: TPlugs extends { tools: unknown[] }
  ? string | BrainPlugToolExecution[]
  : string;
```

the implementation (line 64) uses unconditional union:
```ts
prompt: string | BrainPlugToolExecution[];
```

**backcompat impact**: this is SAFE because:

1. **type broadened is backwards compatible**: prior callers who pass `string` still work — `string` is a subset of `string | BrainPlugToolExecution[]`

2. **runtime behavior unchanged**: the `Array.isArray(askInput.prompt)` check at line 98 ensures string prompts are handled via the `else` branch (line 124-127), same as before

3. **no conditional type complexity**: the unconditional approach is simpler and still type-safe — callers cannot accidentally pass wrong prompt type

4. **rhachet BrainAtom interface constraint**: the `ask` function must conform to rhachet's BrainAtom interface, which may not support conditional prompt types — the unconditional approach is the pragmatic choice

**verdict**: the implementation diverges from blueprint in a way that is MORE permissive, not more restrictive. this is safe for backcompat.

### response_format conditional logic — behavioral analysis

the implementation adds conditional response_format at lines 142-159:

```ts
const hasTools = tools && tools.length > 0;
const wantStructuredOutput = !hasTools || promptIsToolExecutions;
```

**backcompat impact**: this is SAFE because:

1. **prior behavior (no tools)**: `hasTools = false` → `wantStructuredOutput = true` → response_format sent. identical to before.

2. **new behavior (with tools, initial call)**: `hasTools = true` && `promptIsToolExecutions = false` → `wantStructuredOutput = false` → response_format NOT sent. this is NEW behavior for NEW codepath.

3. **new behavior (with tools, continuation)**: `hasTools = true` && `promptIsToolExecutions = true` → `wantStructuredOutput = true` → response_format sent. this is NEW behavior for NEW codepath.

**verified**: prior callers never pass `plugs.tools`, so they always hit codepath #1 with identical behavior.

### BrainOutput shape change — type analysis

the blueprint adds `calls: { tools: BrainPlugToolInvocation[] } | null` to BrainOutput.

**backcompat impact**: this is SAFE because:

1. **rhachet BrainOutput type**: this property is defined by rhachet's BrainOutput class, not by this implementation. the type change comes from rhachet.

2. **prior callers**: did not access `result.calls` (it was undefined). adding it as `null` when no tools are called is backwards compatible — `undefined` and `null` are both falsy.

3. **type inference**: prior callers get `BrainOutput<TOutput, 'atom', BrainPlugs>` which has `calls: null` (no tools → no calls).

## lessons from this review

1. **implementation diverged from blueprint**: the prompt type is unconditional (`string | BrainPlugToolExecution[]`) rather than conditional on TPlugs. this is SAFE because broadened types are backwards compatible.

2. **verified runtime paths**: the `Array.isArray()` check ensures prior callers (who pass strings) hit the same codepath as before.

3. **rhachet contract constraints**: the implementation must conform to rhachet's BrainAtom interface, which may not support conditional types. the unconditional approach is pragmatic.

4. **new codepaths for new functionality**: the tool-related codepaths (lines 99-127, 217-247) only execute when tools are plugged — prior callers never hit these paths.
